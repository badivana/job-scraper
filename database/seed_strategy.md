# Seed Strategy

## Purpose

To define a consistent and maintainable approach for populating the database with initial or reference data (seed data) that supports development, testing, and, where appropriate, production baseline setup (e.g., reference tables, default configuration).

## Overview

Seed data refers to static or semi‑static data that is inserted into the database to make the system functional out of the box (e.g., lookup tables, default roles, configuration values). It is **not** the same as mutable application data (user‑generated content, jobs, resumes). We distinguish three categories:

1. **Reference Data** – Enumerations, status codes, currency codes, country lists, etc. Rarely changes; treated as immutable after initial load.
2. **Configuration Data** – Feature flags, default notification templates, system‑wide parameters. May evolve over time but changes are infrequent and controlled.
3. **Demo/Bootstrap Data** – Sample users, jobs, resumes used for demonstration, testing, or performance‑benchmarking. Loaded only in non‑production environments (local, CI, staging) and never in production unless explicitly approved.

## Types of Seed Data

| Type | Example | Load Where | Update Frequency |
|------|-----------|------------|------------------|
| Reference | `employment_type` (full‑time, part‑time, contract, internship, temporary) | All environments (dev, test, staging, prod) | Very low (only when business adds new types) |
| Reference | `currency` (USD, EUR, GBP, …) | All | Low |
| Configuration | `notification_templates` (welcome email, password reset) | All | Low to medium (updated via admin UI or migration) |
| Configuration | `feature_flags` table (flags for gradual rollout) | All | Medium (changed via release process) |
| Demo | `demo_user` (email: demo@example.com) | Dev, Test, Staging | Never in production; refreshed as needed |
| Demo | Sample job postings (from public sources) | Dev, Test, Staging | Periodically refreshed for realism |
| Benchmark | Large synthetic dataset (e.g., 1M jobs) | Performance test environments only | As needed for load tests |

## Seed Data Management

We store seed data declarations in the `database/seeds/` directory, organized by type:

```
database/
├── seeds/
│   ├── reference/
│   │   ├── employment_type.sql
│   │   ├── currency.sql
│   │   └── …
│   ├── configuration/
│   │   ├── notification_templates.sql
│   │   └── feature_flags.sql
│   └── demo/
│       ├── demo_users.sql
│   └── benchmark/
│       └── jobs_large.sql
```

### Format

- **SQL files** containing `INSERT` statements (or `INSERT … ON CONFLICT DO NOTHING` for idempotency).
- Each file should be **self‑contained** and **order‑independent** where possible; dependencies are managed by naming or by a master script.
- We avoid using `auto_increment` values in inserts; instead we rely on `DEFAULT` (e.g., UUIDs generated by `uuid_generate_v4()`) or explicit UUIDs for reference rows if needed for foreign keys in other seed files.
- For reference data we often use a **lookup table** with a `code` (unique, short) and a `description`. Seed files insert based on the `code`.

### Example: employment_type.sql

```sql
INSERT INTO employment_type (code, label, description)
VALUES
  ('full_time', 'Full‑Time', 'Standard full‑time employment'),
  ('part_time', 'Part‑Time', "Part‑time work (<30 h/week)"),
  ('contract', 'Contract', "Fixed‑term contract"),
  ('internship', 'Internship', "Trainee position"),
  ('temporary', 'Temporary', "Short‑term or temp agency work")
ON CONFLICT (code) DO UPDATE
SET label = EXCLUDED.label,
    description = EXCLUDED.description;
```

We use `ON CONFLICT DO UPDATE` (PostgreSQL `INSERT … ON CONFLICT`) to make the script idempotent: re‑running it will update existing rows to the defined values, ensuring drift is corrected.

## Seed Data Environments

| Environment | Seed Data Loaded |
|-------------|------------------|
| **Local Development** | All reference, configuration, demo, and optionally benchmark data (if developer chooses). |
| **CI / Test** | Reference, configuration, and a minimal demo dataset sufficient for test execution (may be a subset of full demo). |
| **Staging** | Reference, configuration, and a realistic demo dataset (closer to production volume). |
| **Production** | **Only** reference and configuration data. No demo or benchmark data is ever loaded into production. |
| **Performance Test** | Reference, configuration, and a synthetic benchmark dataset scaled to the desired load test size. |

## Automation and Tooling

### Loading Seeds

We provide a script `database/load_seeds.sh` (or a Makefile target) that:

1. Checks the target environment variable (`APP_ENV` or similar) to decide which seed sets to load.
2. Executes the SQL files in a defined order (reference → configuration → demo → benchmark) using `psql` (for PostgreSQL) or the Supabase CLI.
3. Uses `--set ON_ERROR_STOP=on` to abort on any error.
4. Logs success/failure to stdout and optionally to a file for debugging.

**Example invocation:**

```bash
# Load all seeds for local dev
./database/load_seeds.sh env=local

# Load only reference and configuration for CI
./database/load_seeds.sh env=ci
```

### Integration with Migrations

Seed data that is considered part of the **schema baseline** (e.g., a new lookup table added in a migration) should be seeded **within the same migration** using `INSERT … ON CONFLICT DO NOTHING`. This ensures that when the migration runs on any environment, the reference data is present immediately.

For seed data that is independent of schema changes (e.g., adding new demo users), we treat it as a **data deployment** step separate from migrations, executed after migrations but before the application starts.

### Versioning Seed Data

Seed data files are versioned alongside the codebase. When we need to modify existing reference data (e.g., add a new employment type), we:

1. Create a new SQL file (or edit the existing one) with the desired `INSERT … ON CONFLICT DO UPDATE`.
2. Ensure the change is backward compatible (does not break existing references).
3. Include the change in the same pull request that introduces the feature requiring the new data.

We do **not** maintain a separate version number for seed data; its evolution is tracked via Git.

## Security and Sensitivity

- **No Secrets**: Seed data must never contain passwords, API keys, or personal data. Any secrets are managed via environment variables or secret managers (AWS Secrets Manager, GCP Secret Manager, Vault, etc.).
- **Masking**: If we ever need to include realistic‑looking but fake PII for demo purposes, we use clearly synthetic values (e.g., email `@example.com`, phone numbers `555‑0000`) and mark them as non‑production.
- **Access Control**: The SQL files are readable by anyone with repository access, but the execution of seed scripts is limited to environments where the runner has database access (typically CI/CD agents or admins). Production seed runs are strictly prohibited.

## Responsibilities

- **Product Owner / Data Steward**: Determines what reference data is needed and reviews changes to reference data.
- **Backend Engineer**: Creates and maintains seed SQL files, ensures they are idempotent, and integrates them into the deployment flow.
- **DevOps / Platform**: Provides the `load_seeds.sh` script (or equivalent), ensures it runs at the appropriate stage (post‑migration, pre‑start), and logs success/failure.
- **QA / Test Engineer**: Validates that seed data yields expected behavior in test cases (e.g., dropdowns are populated correctly).
- **Security Team**: Audits seed files for accidental inclusion of sensitive data and ensures that encryption/secrets policies are followed for any auxiliary files.
